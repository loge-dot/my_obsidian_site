Github协作

既然是7人团队，并且采用任务驱动的方式进行开发，那么 GitHub 的协作方式需要保证：

✅ 任务清晰可追踪——每个人都知道自己负责什么，避免重复工作或遗漏任务。
✅ 代码管理高效——减少合并冲突，提高开发效率。
✅ 版本控制规范——确保代码提交有迹可循，方便回溯问题。

🚀 最佳协作方案

你的团队可以采用 GitHub Projects + Git Flow 进行任务管理，并结合 Pull Request（PR）+ Code Review 机制保证代码质量。下面是具体方案：

1. 任务管理方式

🔹 使用 GitHub Issues + Projects

GitHub 提供了 Projects（项目看板） 来管理任务进度，每个任务都可以用 Issue（问题） 进行跟踪。

👉 具体操作

  1. 创建 GitHub Project

    • 在 GitHub 仓库，进入 Projects → New Project
    • 选择 Board（看板） 视图，类似于 Jira、Trello

  2. 配置任务状态列

    • 🟡 To Do（待办）
    • 🔵 In Progress（进行中）
    • 🟢 Reviewing（代码审核）
    • ✅ Done（已完成）

  3. 创建 Issue 作为任务

    • 每个开发任务创建一个 Issue（例如 实现用户登录 API）
    • Issue 需要 指派负责人（Assignee）
    • 设定 截止日期（Milestone）
    • 添加 标签（Label），如 bug、feature、enhancement

  4. Issue 与 Project 关联

    • 任务开始时，将 Issue 移动到 In Progress
    • 完成后提交 Pull Request（PR），移动到 Reviewing
    • 通过 Review 后，合并 PR，并将 Issue 关闭，移动到 Done

2. 分支管理策略

🔹 使用 Git Flow

为了保证代码稳定性，建议使用 Git Flow 分支管理策略：

📌 分支结构

  • main（主分支）：稳定代码，仅用于发布版本，不直接开发
  • develop（开发分支）：日常开发在 develop 进行
  • feature-*（功能分支）：新功能开发，从 develop 拉出，完成后合并回 develop
  • hotfix-*（紧急修复分支）：如果 main 出现重大 bug，从 main 拉出 hotfix，修复后合并回 main 和 develop
  • release-*（发布分支）：当 develop 代码稳定时，创建 release 分支进行测试，最终合并到 main

👉 具体流程

  1. 开发新功能

    git checkout develop
git pull origin develop
git checkout -b feature-login

  2. 提交代码

    git add .
git commit -m "feat(auth): 实现 JWT 登录"
git push origin feature-login

  3. 创建 Pull Request（PR）

    • 在 GitHub 上创建 PR，把 feature-login 合并到 develop
    • 指定至少 1-2 名团队成员 Code Review
    • 通过后合并 PR，删除 feature-login

  4. 发布新版本

    git checkout release-1.0
# 进行测试...
git checkout main
git merge release-1.0
git tag v1.0
git push origin main --tags

3. Git 提交规范

🔹 使用标准化的 Commit Message

团队 7 个人协作时，必须让提交信息清晰，建议使用 Conventional Commits 规范：

<type>(<scope>): <subject>

<body>（可选，详细描述）
<footer>（可选，关联 Issue）

常见 type 类型

type 作用
feat ✨ 新功能（如 feat(auth): 实现 JWT 登录）
fix 🐛 Bug 修复（如 fix(api): 修复 Token 过期问题）
docs 📖 文档修改（如 docs(readme): 更新 API 说明）
refactor 🔨 代码重构（如 refactor(db): 优化数据库查询）
test ✅ 测试相关（如 test(user): 增加用户注册单元测试）

示例

git commit -m "feat(auth): 添加 OAuth2.0 登录支持"
git commit -m "fix(api): 修复用户头像加载错误 closes #12"

4. 代码审核（Code Review）

🔹 使用 Pull Request（PR） 进行合并

禁止直接提交 main 或 develop，所有代码都必须通过 PR 合并：

  1. 新功能开发完毕后
    • 由开发者创建 PR，并添加详细描述
    • 例如： 
      实现 OAuth2.0 登录功能，支持 Google、GitHub 登录。

closes #25
  2. 指定 Reviewer
    • 指定 1-2 位团队成员 Code Review
    • 通过后合并到 develop
    • 未通过的 PR 需要修改后重新提交

🔹 Code Review 规则

  • 审查代码风格（是否符合团队规范？）
  • 检查逻辑错误（是否可能有 bug？）
  • 建议优化点（有没有更好的实现方法？）

5. 自动化 & 质量保障

🔹 配置 GitHub Actions 自动化

为了确保代码质量，可以在 GitHub Actions 配置：

  • ✅ 自动运行单元测试
  • ✅ 自动执行 ESLint / Prettier（前端）
  • ✅ 自动进行代码风格检查（Python Black、Flake8）

示例：.github/workflows/test.yml

name: Run Tests

on:
  pull_request:
    branches:
      - develop

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v2

      - name: Install dependencies
        run: npm install  # Python 可换成 pip install -r requirements.txt

      - name: Run tests
        run: npm test  # Python 可换成 pytest

当 PR 创建时，GitHub Actions 会自动执行测试，测试通过才能合并。

6. 进度同步方式

🔹 每周例会 + Standup Meeting

为了保持进度同步，团队可以：

  1. 每周开一次例会
    • 讨论本周任务完成情况
    • 规划下周任务
  2. 每日 Standup Meeting（站会）
    • 每个人汇报：
      • 今天做了什么？
      • 遇到什么问题？
      • 需要什么帮助？

🔹 使用 Notion / 飞书 / Slack 进行沟通

  • GitHub 只能管理代码，但团队沟通需要更顺畅
  • 可以用 Notion / 飞书 / Slack 记录任务进展

🎯 结论

📌 任务管理：使用 GitHub Issues + Projects，每个任务都有 Issue 追踪
📌 分支管理：使用 Git Flow，避免代码冲突
📌 提交规范：采用 Conventional Commits，保证历史清晰
📌 代码审核：强制 Pull Request + Code Review
📌 自动化测试：GitHub Actions 运行单元测试，确保质量
📌 团队沟通：每日 Standup Meeting，每周例会，使用 Notion 记录任务

🚀 这样管理后，团队协作效率一定会大幅提升！
